## Purpose

This document explicitly defines which files within the CA360ยบ portal must follow v6.0 canonical nomenclature and which are exempt, providing audit clarity and preventing interpretation drift.

## v6.0 Canonical Format

```
[ATA]_[PROJECT]_[PROGRAM]_[FAMILY]_[VARIANT]_[VERSION]_[MODEL]_[BLOCK]_[PHASE]_[KNOT]_[AoR]__[SUBJECT]_[CATEGORY]_[TYPE]_[ISSUE-REV]_[STATUS].[EXT]
```

**Key Requirement:** PROJECT=AMPEL360 is **mandatory** in all in-scope filenames.

## In-Scope Files (Must Use v6.0 Format)

### 1. Ledger Artifacts
**Location:** `CAXS/LEDGERS/**/*.md`  
**Rationale:** Ledgers contain authoritative records; canonical naming ensures traceability and uniqueness validation.

**Examples:**
- `85_AMPEL360_SPACET_Q10_BASELINE_PLUS_PR_80_LC09_K09_STK_CEGT__circularity-kpis_REGISTRY_IDX_I01-R01_ACTIVE.md`
- `85_AMPEL360_SPACET_Q10_BASELINE_PLUS_PR_80_LC09_K09_STK_CEGT__esg-reporting_REGISTRY_IDX_I01-R01_ACTIVE.md`

### 2. Deliverable Artifacts
**Location:** `CAXS/**/*_DELIVERABLE_*.md`  
**Rationale:** Deliverables are subject to "one official chain" rule; v6.0 naming enables automated uniqueness checks.

**Examples:**
- `00_AMPEL360_SPACET_Q10_BASELINE_PLUS_PR_00_LC01_K01_STK_CM__ca360-portal-rc-protocol_DELIVERABLE_PROC_I01-R01_ACTIVE.md`
- `42_AMPEL360_SPACET_Q10_BASELINE_PLUS_HW_20_LC03_K03_STK_DAB__ima-architecture_DELIVERABLE_SPEC_I01-R01_ACTIVE.md`

### 3. Evidence Artifacts
**Location:** `CAXS/**/*_EVIDENCE_*.md`  
**Rationale:** Evidence must be traceable to requirements/design; v6.0 naming provides deterministic reference keys.

### 4. Signoff Artifacts
**Location:** `CAXS/**/*_SIGNOFF_*.md`  
**Rationale:** Signoffs are governance-critical; canonical naming ensures audit trail integrity.

### 5. Registry Artifacts  
**Location:** `CAXS/**/*_REGISTRY_*.md`  
**Rationale:** Registries are SSOT artifacts; v6.0 naming enables automated indexing and validation.

### 6. Export Control Artifacts
**Location:** `CAXS/**/*_EXPORT_CONTROL_*.md`  
**Rationale:** Regulatory compliance requires deterministic naming and traceability.

## Out-of-Scope Files (v6.0 Optional)

### 1. README Files
**Pattern:** `**/README.md`  
**Rationale:** READMEs are descriptive documentation, not governed artifacts. Human-readable names preferred.

**Examples:**
- `CAXS/README.md`
- `CAXS/AoR/README.md`
- `CAXS/KNOTS/README.md`

### 2. Template Files
**Pattern:** `**/templates/**/*_template.md`  
**Rationale:** Templates are infrastructure, not artifacts. Descriptive names aid usability.

**Examples:**
- `CAXS/INFRASTRUCTURE/templates/aor_ssot_pack/roadmap_registry_template.md`
- `CAXS/INFRASTRUCTURE/templates/task_template.md`

### 3. AoR Internal Registries
**Pattern:** `CAXS/AoR/STK_*/{roadmap_registry,task_registry,evidence_index,signoff_index}.md`  
**Rationale:** These are operational registries managed by AoRs, not cross-program artifacts. Simplicity preferred for frequent updates.

**Note:** These MAY be renamed to v6.0 format in future if cross-AoR traceability requirements emerge.

### 4. Configuration Files
**Pattern:** `configs/**/*.{json,yml,yaml}`  
**Rationale:** Configuration files use standard naming conventions (JSON/YAML). v6.0 format not applicable.

**Examples:**
- `configs/nomenclature/v6/vocabulary.json`
- `CAXS/INFRASTRUCTURE/validators/exceptions.yml`

### 5. Infrastructure Documentation
**Pattern:** `CAXS/INFRASTRUCTURE/**/*.md` (non-artifact files)  
**Rationale:** Portal contract files and infrastructure docs are not governed artifacts.

**Examples:**
- `CAXS/INFRASTRUCTURE/portal_contract_files_v1.0.md`
- `CAXS/INFRASTRUCTURE/indexes/portal_entrypoints_index.md`
- `CAXS/INFRASTRUCTURE/validators/one_official_chain_uniqueness.md`

**Exception:** If infrastructure docs are packaged as DELIVERABLE for external release, they MUST use v6.0 format.

### 6. Reports and Validation
**Pattern:** `CAXS/REPORTS/**/*_{template,report}.md`  
**Rationale:** Templates and filled reports are operational artifacts, not governed deliverables.

**Examples:**
- `CAXS/REPORTS/validation/ca360_portal_validation_report.md`
- `CAXS/REPORTS/validation/release_evidence_pack_template.md`

### 7. Deprecated Files
**Pattern:** `CAXS/*DEPRECATED*.md`  
**Rationale:** Files marked for removal; naming compliance not required.

## Enforcement

### CI Gates
The portal gates workflow validates v6.0 nomenclature for:
- All files in `CAXS/LEDGERS/` (excluding README.md)
- All files matching `*_DELIVERABLE_*.md`
- All files matching `*_EVIDENCE_*.md`
- All files matching `*_SIGNOFF_*.md`
- All files matching `*_REGISTRY_*.md` (excluding internal AoR registries)
- All files matching `*_EXPORT_CONTROL_*.md`

### Scope Validation
Files not matching in-scope patterns are excluded from v6.0 validation.

### Audit Trail
This scoping decision is recorded in:
- This document (v6.0_nomenclature_scope.md)
- Portal contract file list (portal_contract_files_v1.0.md)
- Release evidence packs (for each RC/release)

## Rationale for Scoping Decision

### Why In-Scope?
Files subject to governance rules (one official chain, traceability, signoffs) MUST use v6.0 naming to enable automated validation and prevent drift.

### Why Out-of-Scope?
Infrastructure files, templates, and operational registries benefit from human-readable naming. Forcing v6.0 format would reduce usability without governance benefit.

### Certifiable-Grade Consistency
By explicitly documenting the scope, we maintain:
- **Consistency:** All governed artifacts follow same naming
- **Clarity:** Auditors can verify scope compliance
- **Flexibility:** Infrastructure can evolve without breaking governance
